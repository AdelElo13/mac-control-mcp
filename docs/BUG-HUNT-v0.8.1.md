# mac-control-mcp — Verified Bug Report

Method: 16 parallel reviewers over all 55 Swift files (~15.6K LOC) + 5 cross-cutting lenses; every finding adversarially verified by 3 independent skeptic agents (refute / reproduce / design-intent). A finding is listed only if ≥2 of 3 verifiers independently confirmed it real and reachable. Several crashes were reproduced empirically against `.build/debug/mac-control-mcp` (exit 133).

Total confirmed: 57  |  critical 4 · high 19 · medium 30 · low 4


## CRITICAL (4)

### 1. MCPProtocol.swift:93
**JSONValue.intValue does an unchecked Int(Double) conversion that traps and crashes the whole server on any large-magnitude numeric argument**

In JSONValue.intValue the .number branch only guards that the value is integral (value.rounded() == value) and then force-converts with Int(value). For any JSON number whose magnitude exceeds Int.max (e.g. 1e300, or even a plain large integer literal like 99999999999999999999 which JSONDecoder parses as a Double ~1e20), value.rounded() == value is still true, so it falls through to Int(value) which is a runtime trap ('Double value cannot be converted to Int because the result would be greater than Int.max'). This aborts the entire process (SIGTRAP, exit 133), not just the one request. It is reachable from untrusted client JSON via every int-typed argument: parsePID(arguments["pid"]) (Tools.swift:796 calls value?.intValue before its own <= Int32.max bounds check ever runs), max_depth (Tools.swift:480), and every other intValue caller. A single tools/call request kills the server, so it is a trivial denial-of-service. Fix: range-check before converting, e.g. guard value >= Double(Int.min) && value < Double(Int.max) else { return nil }, or use Int(exactly:).

_Trigger:_ Client sends {"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"list_elements","arguments":{"pid":99999999999999999999}}}. Verified end-to-end against .build/debug/mac-control-mcp: the process aborts with 'Fatal error: Double value cannot be converted to Int because the result would be greater than Int.max' and exit code 133, taking down the whole MCP server. Same crash reproduced via {"pid":1e300} and via {"pid":123,"max_depth":1e19}.

### 2. MCPProtocol.swift:244
**Content-Length framer computes bodyStart + contentLength without overflow checking, trapping on an oversized Content-Length header**

In StdioMessageFramer.popContentLengthFramed, parseContentLength accepts any Content-Length value that fits in Int and is >= 0 (line 312 allows up to Int.max). The framer then computes let bodyEnd = bodyStart + contentLength (line 244) with default checked arithmetic. Since bodyStart is a positive header offset and contentLength can be as large as Int.max, this addition overflows Int, which is a runtime trap in Swift and aborts the whole process (exit 133) before the guard buffer.count >= bodyEnd on the next line can reject it. Because popMessage still accepts Content-Length framing on input for backward compatibility, any client (or malformed probe) can send a single frame beginning with a huge Content-Length header and crash the server. Fix: use bodyStart.addingReportingOverflow(contentLength) (or a UInt64/overflow-safe compare) and treat overflow as malformed, and/or cap Content-Length to a sane maximum.

_Trigger:_ Client writes the raw bytes 'Content-Length: 9223372036854775807\r\n\r\n{}' to stdin. Verified end-to-end against .build/debug/mac-control-mcp: a valid small Content-Length frame returns normally (exit 0), but the Int.max Content-Length frame crashes the server with exit code 133 (arithmetic overflow trap), a denial-of-service from one input frame.

### 3. OsascriptRunner.swift:28
**OsascriptRunner.run reads pipes only AFTER waitUntilExit and has no timeout — deadlocks/hangs the whole server (hits browser_visible_text, browser_dom_tree on any large page)**

Unlike ProcessRunner.run, which attaches a watchdog that terminates the child at a deadline, OsascriptRunner.run calls process.waitUntilExit() synchronously with no timeout and no watchdog. osascript commonly blocks indefinitely: on the first Apple Events / Automation TCC consent dialog (e.g. 'tell application "System Events"' from SystemController.setDarkMode, PowerController.restart/shutdown/logout, NotificationCenterController.toggle, AudioController.setMicMute, UndoController.revert), or when the target app is unresponsive. Because this synchronous call runs inside an actor method on a Swift-concurrency cooperative thread, the hang wedges that actor permanently and consumes a pool thread; several concurrent hung calls can starve the whole server. It also has the same read-after-wait pipe pattern as ProcessRunner (deadlock if the script prints >64KB).

_Trigger:_ Fresh install, no Automation permission yet. Client calls set_dark_mode → osascript triggers the 'wants to control System Events' consent dialog → user is away / server runs unattended → the MCP tool call never returns, SystemController actor is blocked forever, and every subsequent set_dark_mode/set_volume call queues behind it.

### 4. Tests/Phase6ToolsTests.swift:30
**Phase 6 tests wipe and overwrite the user's real ~/.mac-control-mcp/permissions.json on every `swift test` run (silent data loss).**

PermissionStore is a singleton actor whose storePath is hard-coded to ~/.mac-control-mcp/permissions.json (PermissionTiers.swift:120-126) with no test-mode redirect. `_resetForTesting()` (PermissionTiers.swift:294-297) sets grants=[:] and loaded=true, so the real file is never loaded into memory. Every subsequent grant()/deny()/revoke() in the tests calls persist(), which rewrites the REAL file with only the in-memory (test) entries. Net effect of grantCheckListRevoke, denyBeatsGrant, expiredGrantsRejected, requestAccessRoundTrip and denyAccessToolRoundTrip: all real user grants are destroyed, and test junk is persisted (com.test.deny is a 30-day deny entry, com.test.roundtrip a 300s grant). I verified the live file currently contains a real grant: [{"tier":"full","bundleId":"com.apple.Safari",...}] — the next test run deletes it.

_Trigger:_ User has granted com.apple.Safari full tier (present in the real permissions.json today). Developer runs `swift test`. Phase6ToolsTests resets the in-memory store without loading the file, grants/revokes test bundles, and each mutation persists — the file ends up containing only leftover test entries (e.g. a 30-day deny for com.test.deny). The Safari grant is gone; production MCP clients silently lose authorization and must re-request, with no indication tests caused it.


## HIGH (19)

### 5. AXObserverBridge.swift:195
**Race in waitForNotification: cleanup closure is installed after the observer is already live, so an early-firing notification permanently leaks the observer and leaves its runloop source attached forever**

The AX notification can be delivered as soon as CFRunLoopAddSource (step 4, line 191) executes, because the bridge's dedicated runloop thread is already running. The cleanup closure that removes the source, unregisters the notification, and releases the passRetained refcon is only assigned at step 5 (line 195). If the callback fires in that window, AXObserverWait.resume() runs with cleanup == nil: it sets fired=true, calls cleanup?() as a no-op, and resumes the continuation. The subsequent assignment of box.cleanup is then never invoked by anyone — the retained AXObserverWait leaks, AXObserverRemoveNotification is never called, and the observer's runloop source stays in the bridge runloop for the life of the process, waking the runloop on every future notification from that element. Additionally, the write to box.cleanup happens on the caller's thread without taking box's NSLock, while resume() reads cleanup under the lock from the runloop thread — an unsynchronized cross-thread write/read on the same property (a data race TSan would flag). The fix is to install cleanup (or at least take the lock) before AXObserverAddNotification/CFRunLoopAddSource make the callback reachable, and have resume() run cleanup if it was installed late.

_Trigger:_ Agent calls wait_for_ax_notification with notification=AXValueChanged on an element that is updating every frame (e.g. a progress bar or a text field being typed into). The notification is delivered on the bridge runloop thread microseconds after CFRunLoopAddSource, before line 195 runs on the caller thread. resume() fires with cleanup nil; the wait returns 'fired' correctly, but the AXObserver registration, its runloop source, and the retained AXObserverWait are leaked permanently. Over a long-lived server session with many waits on chatty notifications, leaked observers accumulate and the bridge runloop is continuously woken by stale sources.

### 6. AXSnapshotController.swift:72
**ax_snapshot_capture walks the AX tree with no wall-clock deadline and no node cap, so a capture on a large app blocks the server for tens of seconds to minutes**

AXSnapshotController.walk recurses the full AX tree bounded only by maxDepth (tool layer clamps to 12 default / 32 max in Tools+V2Phase9.swift:261) and the CFHash visited-set. There is no time budget and no node cap. Every node costs 5 blocking AX IPC round trips (role, title, value, position, size) plus one for children. AccessibilityController deliberately added a 5 s / 5000-node budget to treeWalk/findElements/queryElements after this exact hang class was 'reported in v0.2.1 testing' (its own comments say the budget 'matches the 5 s / 5000-node budget used elsewhere'), but this newer controller was written without it. Since AXSnapshotController is an actor and the walk is fully synchronous, the tool call blocks until the walk finishes; the MCP client typically times out and gives up first.

_Trigger:_ Client calls ax_snapshot_capture with pid of Finder showing a large list view, Mail with a full message list, or Logic Pro, and max_depth=32 (accepted by the clamp). The walk performs hundreds of thousands of AX IPC calls with no abort condition; the tool call hangs far past any client timeout, exactly the failure mode the 5 s deadline was introduced to fix in AccessibilityController.

### 7. AccessibilityController.swift:147
**enableManualAccessibility permanently sets AXEnhancedUserInterface=true on every app it touches, which is documented to break AX window positioning — subsequent move_window/resize_window calls silently misbehave**

Confirmed reachable: WindowController.moveWindow/resizeWindow (lines 219-234) route through window()→axWindows()→enableManualAccessibility() (line 33-43), which permanently sets AXEnhancedUserInterface=true on the target pid and caches it, with no code anywhere toggling it off before the kAXPositionAttribute/kAXSizeAttribute writes (grep found only kCFBooleanTrue for that attribute; the only kCFBooleanFalse uses are for minimized/fullscreen). Enabling AXEnhancedUserInterface while it was off is the exact condition the yabai/Rectangle/Amethyst window-manager community documents as breaking AX window positioning (windows animate/land imprecisely), and here AXUIElementSetAttributeValue still returns .success so callSMoveWindow (Tools+V2Phase4.swift:95-105) reports 'Window moved.'/success:true — a silent misbehavior exactly as claimed. The source comment's assertion that setting it is a 'no-op on non-Electron apps' is false (that is the root cause), though the finding slightly overstates 'every app': breakage manifests on standard AppKit apps honoring enhanced-UI semantics, which are nonetheless common move/resize targets, so real user harm holds. Medium because it silently degrades positioning precision and misreports success rather than crashing or losing data.

### 8. AppleAppsController.swift:165
**mail_send with send_now=false neither saves a draft nor shows a window, yet reports success 'draft opened'**

The mail_send tool (Tools+V2Phase8.swift callMailSend) promises for send_now=false to 'save as draft and open Mail for review'. But sendMail() always creates the outgoing message with visible:false and, when sendNow is false, only runs `activate` — it never calls `save` and never sets the window visible. _Trigger:_ Call mail_send(to:"x@y.com", subject:"Hi", body:"draft me", send_now:false). Mail comes to the front but shows no compose window and no Drafts entry; the tool returns ok:true 'draft opened'. The user has nothing to review and the email is gone.

### 9. AppleAppsController.swift:57
**iMessage recipient escaping omits backslash, breaking/allowing AppleScript-literal injection in imessage_send**

In sendMessage(to:body:) the recipient is escaped with only a quote replacement — `escTo = to.replacingOccurrences(of: "\"", with: "\\\"")` — and NEVER escapes backslash. It is then interpolated raw into an AppleScript string literal: `set targetBuddy to buddy "\(escTo)" of targetService`. Because backslash is AppleScript's own escape character, a backslash in `to` survives unescaped and corrupts the literal. This is a clear regression: the very next lines escape the body correctly (`escBody` does `\` -> `\\` FIRST, then `\"` -> `\\\"`), proving the intended rule was simply forgotten for the recipient. Result: the closing quote the template supplies gets turned into an escaped quote, so the string never terminates — the message is silently not sent and the tool returns a confusing AppleScript compile error instead of a clean validation failure.

_Trigger:_ Call imessage_send with to = `alice@example.com\` (trailing backslash) or any handle containing `\`. The generated script becomes `buddy "alice@example.com\" of targetService`, where `\"` is an escaped quote — the literal is never closed, osascript fails to compile, and sendMessage returns ok:false with a garbled stderr while the message is never delivered. An agent building the recipient from untrusted text (a name/handle scraped from a page) triggers this without any user intent.

### 10. AppleNativeController.swift:218

invokeAppIntent forwards user input to the shortcuts CLI as `--input <value>`. That flag does not exist: `shortcuts run` only accepts `-i/--input-path <path>` (and `--output-path`/`--output-type`). ### 11. BrowserController.swift:278
**BrowserController osascript calls deadlock (infinite hang) when AppleScript stdout exceeds the pipe buffer**

runOsascript delegates to the shared OsascriptRunner.run, which calls process.waitUntilExit() BEFORE draining the stdout pipe (it only calls readDataToEndOfFile after the process has exited) and sets no timeout. When osascript writes more to stdout than the OS pipe buffer holds (~16-64 KB), the child blocks on write(), never terminates, and waitUntilExit() blocks the calling thread forever. Because BrowserController is an actor, the hung call permanently wedges the actor and every subsequent browser tool call serializes behind it. evalJS wraps its result in JSON.stringify and returns it through stdout; browser_dom_tree and browser_visible_text (BrowserDOMController -> evalJS -> runOsascript) serialize an entire page's DOM / visible text with no size cap, and listTabs concatenates title+URL for every tab. Any of these can exceed 64 KB. Unlike ProcessRunner (which at least has a 10s watchdog), OsascriptRunner has neither a watchdog nor a timeout, so the hang is unbounded and requires a server restart to clear.

_Trigger:_ Call browser_visible_text or browser_dom_tree on a content-heavy page (large article, web app) whose serialized output exceeds ~64 KB. osascript fills the stdout pipe, blocks on write(), never exits; waitUntilExit() never returns; the tool call hangs indefinitely with no timeout, wedging the BrowserController actor until the MCP server is restarted.

### 12. GroundingController.swift:182

### 16. ProcessRunner.swift:52
**ProcessRunner.run reads stdout/stderr only after waitUntilExit — children writing more than the 64KB pipe buffer deadlock until the watchdog SIGTERMs them, producing a false failure with truncated output**

Same read-after-wait ordering as OsascriptRunner: stdout/stderr pipes are drained only after process.waitUntilExit() returns. A child that writes more than the ~64KB pipe capacity blocks in write(2); the parent blocks in waitUntilExit; the only escape is the watchdog, which after `timeout` (default 10s) sends terminate() (SIGTERM). The tool call therefore burns the entire timeout, the child is killed mid-output, terminationStatus is nonzero, and callers report a spurious error (r.ok == false) even though the command was succeeding — with stdout truncated to whatever fit in the pipe. Additionally, terminate() is only SIGTERM: a child that ignores SIGTERM leaves waitUntilExit blocked forever (the watchdog's while-loop returns after the first terminate()), reintroducing the unbounded hang. ProcessRunner backs 30+ tools (HardwareController, SystemInfoController, ShortcutsController, AudioController, VoiceController's say/screencapture, etc.), so any large-output CLI (e.g. `shortcuts list` on a machine with many shortcuts, or a verbose stderr from a brew-installed helper) hits this.

_Trigger:_ list_shortcuts on a Mac where `shortcuts list` outputs >64KB (hundreds of shortcuts / long names): the child blocks writing at 64KB, ProcessRunner blocks in waitUntilExit, the watchdog SIGTERMs it at 10s, and the tool returns a failure with truncated stdout after a 10-second stall — even though the command itself was fine. If the child ignores SIGTERM, the tool call (and the whole single-loop MCP server) hangs indefinitely.

### 18. Tools+V2Phase3.swift:331
**poll_interval_ms has no upper clamp: values > ~1.8e13 overflow-trap in UInt64 multiply (crash), smaller large values defeat the 60s timeout clamp (multi-day hang)**

wait_for_element (Tools+V2Phase3.swift:295/331), wait_for_window (Tools+V2Phase5.swift:544/563), wait_for_app (Phase5:584/618), and wait_for_file_dialog (Phase5:634/667) all parse `let intervalMs = max(arguments["poll_interval_ms"]?.intValue ?? 200, 50)` — lower clamp only — then execute `try await Task.sleep(nanoseconds: UInt64(intervalMs) * 1_000_000)`. Two failure modes: (1) intervalMs > UInt64.max/1_000_000 (~18.4 trillion) makes the multiplication trap (verified empirically: exit 133 overflow trap), crashing the server; a client that confuses ms with epoch-microseconds (~1.7e15) hits this. (2) Any large non-overflowing value bypasses the carefully clamped timeout_seconds (0.1..60): the deadline is only checked BEFORE the sleep, so poll_interval_ms=600000 sleeps 10 minutes and 1e9 sleeps ~11.5 days on the first iteration while the tool call stays pending.

_Trigger:_ wait_for_element with {"pid": 1234, "poll_interval_ms": 1700000000000000} (epoch-microseconds confusion) -> UInt64(1.7e15)*1e6 overflows UInt64 -> arithmetic overflow trap -> server dies. Alternatively poll_interval_ms=3600000 -> tool ignores its documented 60s max timeout and blocks the call for an hour.

### 19. Tools+V2Phase5.swift:563
**Unclamped poll_interval_ms causes UInt64 overflow trap (whole-server crash) or sleeps that vastly exceed the tool's own timeout in wait_for_window, wait_for_app, and wait_for_file_dialog.**

All three polling wait tools compute `intervalMs = max(arguments["poll_interval_ms"]?.intValue ?? 250, 50)` with a lower clamp only, then call `Task.sleep(nanoseconds: UInt64(intervalMs) * 1_000_000)`. Swift arithmetic traps on overflow, so any intervalMs above ~18_446_744_073_709 (UInt64.max / 1e6) aborts the entire MCP server process. JSONValue.intValue (MCPProtocol.swift:89-99) happily parses both large JSON numbers and numeric strings like "100000000000000000" into Int, so this value is fully client-controllable. Even below the overflow threshold, a large-but-valid interval (e.g. 86_400_000 = 24h) makes the first sleep run for a day even though timeout_seconds is clamped to a 60s max — the deadline is only checked between sleeps. Same pattern at lines 563 (wait_for_window), 618 (wait_for_app), and 667 (wait_for_file_dialog). Fix: clamp intervalMs to an upper bound (e.g. min(intervalMs, 5000)) and/or cap each sleep at the remaining time to deadline.

_Trigger:_ Client calls wait_for_window with {"pid": 500, "poll_interval_ms": "100000000000000000"} (1e17, parses fine into Int). No window matches, so the loop reaches Task.sleep: UInt64(1e17) * 1_000_000 = 1e23 > UInt64.max → arithmetic overflow → fatal trap → the mac-control-mcp process dies, killing every other in-flight tool call. Alternatively {"poll_interval_ms": 86400000} with timeout_seconds 5 blocks the call for 24 hours instead of returning the timeout error after 5s.

### 22. VoiceController.swift:147
**text_to_speech speak mode deallocates AVSpeechSynthesizer immediately, so nothing is ever spoken while the tool reports ok:true**

### 23. VoiceController.swift:93
**speechToText keeps no strong reference to the SFSpeechRecognizer during recognition and its continuation has no timeout, so the tool call can hang forever**
